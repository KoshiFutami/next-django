# 0002. showcase バックエンドに薄い DDD を採用する

日付: 2026-04-03

## 状態

Accepted

## 文脈（Context）

- Django で API を素早く立ち上げつつ、ドメインルールが増えたときに肥大化しにくい境界が欲しい。
- いきなりフル DDD（イベントソーシング、複雑な集約設計など）を入れると、MVP 段階でコストが見合わない。

## 決定（Decision）

- `showcase` バックエンドは **薄い DDD** とする。
  - **ドメイン**: `showcase/domain/` — ルールが複雑になったところから厚くする。最初からすべてをドメインモデル化しない。
  - **リポジトリ**: `domain/repositories.py`（Protocol）と `infrastructure/` の実装で永続化を隠す。
  - **ユースケース**: `application/` — オーケストレーションを薄く保つ。
  - **HTTP**: `interface/` — view・URL・JSON 整形。View にビジネスロジックを溜めない方針を基本とする。

## 結果（Consequences）

- レイヤの責務が説明しやすく、テストや差し替えの単位が取りやすい。
- 「薄い」ため、複雑なドメインが来た段階で **集約・値オブジェクト・ドメインイベント** などを段階的に足す必要が出る可能性がある。
- チーム内で「どこまでを domain に書くか」の線引きは、必要に応じて別 ADR やコードレビューで揃える。
