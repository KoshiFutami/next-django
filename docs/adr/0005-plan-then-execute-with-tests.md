# 0005. エージェント作業で計画と実行を分離し完了はテストで検証する

日付: 2026-04-03

## 状態

Accepted

## 文脈（Context）

- コーディングエージェントは**ゴール指向**で、スコープ外を壊してでも直近の目標を達成しようとすることがある。
- **ワンショット問題**として、複数機能を一度に実装しようとして失敗・手戻りが増えやすい。
- 「完了」を宣言しがちだが、**実際には `pytest` や API 疎通を通していない**ことがある（[テストによる完了の検証](https://nyosegawa.com/posts/harness-engineering-best-practices-2026/#4%3A-%E8%A8%88%E7%94%BB%E3%81%A8%E5%AE%9F%E8%A1%8C%E3%82%92%E5%88%86%E9%9B%A2%E3%81%99%E3%82%8B)）。
- Cloudflare Boris Tane の言うとおり、**計画と実行の分離**は無駄な探索を減らし、アーキテクチャのコントロールを保ちやすい（同セクション・計画フェーズ）。

## 決定（Decision）

- **計画 →（必要なら人間の確認）→ 実行**の順を基本とする。IDE に Plan Mode や読み取り専用の計画フェーズがあれば、**大きめの変更や新規 API の前に**使う。
- **タスク粒度は小さく**：原則 **一度に一つの縦割り（1 ユースケースや 1 エンドポイント単位など）**に留め、完了してから次へ進む。
- **完了の定義**に **`make test`（および変更に応じた `make lint-backend`）が通ること**を含める。E2E や手動 API 確認がタスクに含まれる場合は、その手順を明示してから「完了」とする。

## 結果（Consequences）

- 良い影響: 手戻りと巨大 diff が減り、レビューと CI が意味を持ちやすい。
- 負債: 小さなタイポ修正まで毎回計画書にするのは重い。**変更がレイヤ境界・契約・セキュリティに触れるとき**ほど本 ADR を意識する。
- フォローアップ: Claude Code の **Stop フックでテスト強制**は別タスク（無限ループ防止の設計が必要）。[セクション4](https://nyosegawa.com/posts/harness-engineering-best-practices-2026/#4%3A-%E8%A8%88%E7%94%BB%E3%81%A8%E5%AE%9F%E8%A1%8C%E3%82%92%E5%88%86%E9%9B%A2%E3%81%99%E3%82%8B) 参照。
