# 0001. ADR を採用する

日付: 2026-04-03

## 状態

Accepted

## 文脈（Context）

- モノレポでバックエンド・フロント・運用ルールが増えるにつれ、設計判断の理由が会話や PR に散らばりやすい。
- 後から参加する人や将来の自分が「なぜこの構成か」を短時間で把握できる仕組みが欲しい。

## 決定（Decision）

- 重要なアーキテクチャ判断は `docs/adr/` に ADR として記録する。
- 形式は [template.md](template.md) に従い、番号は時系列で付与する。

## 結果（Consequences）

- 判断の履歴が検索しやすくなる。
- 軽微な実装詳細まで ADR にしないと運用が重くなるため、**影響範囲が大きい判断に限定**する。
